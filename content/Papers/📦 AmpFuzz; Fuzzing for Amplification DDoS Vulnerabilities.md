## Abstract

- 문제점
	- 증폭 DDoS 공격은 심각한 위협이며 최근 Tbps급 규모까지 증가했다.
	- 기존 증폭 취약점 발견 방®법은 수동 분석 또는 사후 분석에 의존한다.
- 해결책: **AmpFuzzrr**
	- UDP 서비스에서 *자동으로* 증폭 벡터를 탕ㅁ지하는 첫 체계적인 방법이다.
	- 프로토콜 독립적(protocol-agnostic)으로, 특정 프로토콜에 한정되지 않고 다양한 UDP 서비스 분석이 가능하다.
	- UDP-aware 퍼징 기법을 적용하여 퍼징 성능을 최적화하고 탐지율을 향상한다.
- 실험 결과
	- 28개의 Debian 네트워크 서비스 테스트
	- 7개의 기존 증폭 취약점 재발견, 6개의 새로운 증폭 취약점 발견

## 1. Introduction

 1. 배경: 증폭 DDoS 공격의 심각성
	 - 증폭 DDoS 공격은 가장 강력한 volumetric DDoS 공격 전략 중 하나이다.
	 - UDP 기반 서비스 중 클라이언트의 IP 주소를 검증하지 않는 서비스들이 존재하며, 이를 악용하면 공격자가 출발지 IP를 spoofing하여 피해자의 IP로 응답을 유도할 수 있다.
	 - 이 방식으로 인해 PayPal, Spotify, Twitter, Reddit, eBay 같은 주요 인터넷 서비스가 일시적을 마비되었으며, 2021년에는 2.4Tbps 규모의 공격이 발생하기도 했다.
	 - 새로운 증폭 벡터가 발견될 때마다 기록적인 DDoS 공격이 발생한다:
		 - e.g. Memcached 증폭 벡터가 발견된 직후, 공격자들이 이를 악용하여 대규모 공격을 수행.
		 - e.g. RIPv1(라우팅 프로토콜) 증폭 벡터가 발견된 후, 해당 프로토콜을 이용한 공격이 급증.


2.  기존 증폭 취약점 탐색 방식의 문제점
	- 현재까지 자동화된 새로운 증폭 벡터 탐색 방법이 존재하지 않는다.
	- 대부분의 증폭 취약점은 공격자가 먹저 찾아내며, 연구들은 사후 분석(postmortem analysis)을 통해 대응하는 상황이다.
	- 공격자들은 새로운 증폭 취약점을 발견하면 빠르게 공유하고 공격에 활용하는 반면, 방어자들은 대응이 늦어지는 문제가 발생한다.
	- 이로 인해 네트워크 운영자 및 DDoS 방어 솔루션이 사전에 대응할 수 없고 항상 수동적으로 방어하는 상황이 반복된다.


3. 증폭 취약점 자동 탐색의 필요성
	자동화된 증폭 취약점 탐색은 여러 가지 이점을 제공할 수 있다:
	- 조기 발견(Early Discovery)
	    - 공격이 발생하기 전에 증폭 벡터를 찾아내어 패치하거나 보안 조치를 취할 수 있다.
	    - 과거에도 일부 프로토콜의 개선 및 대규모 취약점 공개(disclosure)가 서비스 보안을 강화하는 데 효과적이었다.
	- 공격 탐지 및 모니터링 강화
	    - 새로운 증폭 벡터를 조기에 발견하면, DDoS 공격 탐지를 위한 Honeypot을 구축하여 실시간 모니터링이 가능하다.
	- DDoS 방어 전략 개선
	    - 자동화된 증폭 취약점 탐색을 통해 DDoS 방어 시스템에 새로운 공격 패턴을 적용하고, 공격이 발생하기 전에 필터링할 수 있다.
    - 이를 통해 기존의 수동적 방어 방식에서 벗어나, 사전 예방적 대응이 가능해진다.


4. 증폭 취약점 탐색의 난제
	==프로토콜 다양성==: 다양한 네트워크 프로토콜이 존재하며, 일부 프로토콜은 공식적인 명세가 부족하다.
- UDP 기반 서비스의 특성:
    - 서비스가 응답 가능한 상태인지 판단하는 것이 어렵다.
    - 서비스가 요청을 완전히 처리했는지 확인하는 것이 외부에서 관찰하기 어렵다.
- 증폭 취약점의 특수성:
    - 일반적인 버그(e.g. 프로그램 충돌) 탐색과 달리, 증폭 취약점은 증폭 계수(Amplification Factor)를 극대화하는 것이 핵심이다.
    - 단순히 취약점이 존재하는지를 확인하는 것이 아니라, *최대한 높은 증폭률을 유도할 수 있는지* 가 중요하다.


5. AMPFUZZ의 기여
	이 연구에서는 최초로 체계적인 증폭 취약점 탐색 방법을 제안한다.
	
	- AMPFUZZ: 프로토콜 독립적인(Protocol-Agnostic) 증폭 취약점 탐색 도구
		- UDP 기반 증폭 취약점 탐색을 자동화.
		- 기존처럼 특정 프로토콜만 분석하는 것이 아니라, 어떤 네트워크 서비스에도 적용할 수 있는 방식.
	
	- UDP-aware 퍼징 기법 도입
		- Greybox Fuzzing) + UDP 인식 기술을 결합하여 기존 퍼징 기법보다 효율적으로 증폭 취약점 탐색 가능하다.
		- UDP 특성을 반영하여, 서비스가 요청을 받을 준비가 된 시점과 응답을 보낼 시점을 감지할 수 있도록 최적화했다.
	
	- 효과적인 증폭 탐색 기법 도입
		- 퍼징을 수행할 때, 단순히 모든 요청을 탐색하는 것이 아니라 "증폭 가능성이 높은 요청 경로"를 우선적으로 테스트하도록 설계되었다.
		- 이를 위해 Static Analysis + 경량화된 동적 계측(Lightweight Instrumentation)을 결합하여 서비스 동작을 효율적으로 분석한다.
	
	- 불필요한 테스트 제거
		- 응답이 없는 테스트 케이스는 조기에 종료하여 퍼징 리소스를 낭비하지 않도록 한다.
		- 기존 방식은 타임아웃 동안 기다려야 했으나, AMPFUZZ는 불필요한 테스트를 사전에 차단하여 성능을 향상한다.
	
	- 증폭 계수 극대화를 위한 변이(Mutation) 전략 적용
		- 퍼징 과정에서 증폭 효과가 큰 요청을 우선적으로 탐색하는 변이 기법을 추가하여, 높은 증폭률을 유도하는 취약점을 빠르게 찾는다.
	
	- DDoS 허니팟(Honeypot)과의 통합
		- AMPFUZZ에서 발견한 증폭 취약점을 기반으로 자동으로 허니팟을 설정할 수 있도록 확장한다.
		- 기존에는 전문가가 직접 공격 요청을 분석하고 허니팟을 구축해야 했으나, AMPFUZZ는 이 과정을 자동화하여 새로운 증폭 공격을 실시간으로 탐지하고 분석 가능하다.

## 2. Background


### 2.1 Fuzzing

- 퍼징은 소프트웨어 취약점을 발견하는 인기 있는 기법으로, 무작위 입력을 프로그램에 전달하고 비정상적인 동작을 감지하는 방식이다.
- 전통적으로 program crashes 탐색에 사용되었지만, 최근에는 performance issues 및 메모리 할당 문제를 찾는 데도 활용된다.

| 퍼징 유형            | 특징                                                        | 장점                         | 단점                                        |
| ---------------- | --------------------------------------------------------- | -------------------------- | ----------------------------------------- |
| Blackbox Fuzzing | 프로그램의 내부 구조를 고려하지 않고 무작위 입력을 생성한다.                        | 빠르게 다양한 입력을 테스트한다.         | 구조화된 입력을 요구하는 프로그램에서는 비효율적이다.             |
| Whitebox Fuzzing | 프로그램을 정적 분석 및 symbolic execution을 통해 내부 제약 조건을 파악한다.      | 새로운 코드 경로로 탐색이 가능하다.       | 실행 속도가 느리고, path explosion 문제가 발생한다.      |
| Greybox Fuzzing  | 블랙박스처럼 빠르게 테스트하면서도, 실행 시 경량 계측을 통해 내부 정보를 활용하여 효율성을 증대한다. | 블랙박스보다 효과적이고, 화이트박스보다 빠르다. | 특정한 취약점 탐색이 어렵고, 고도로 구조화된 입력이 필요한 경우 어렵다. |

- Greybox Fuzzing이 가장 널리 사용된다.
	- 프로그램실행 중에 ==경량 계측==(Runtime lightweight instrumentation)을 추가하여 실행 정보를 피드백으로 활용한다.
	- 코드 커버리지를 최대화하거나 특정한 프로그램 경로를 목표로 설정 가능하다.


### 2.1 Amplification Attacks

- 증폭 공격은 대규모 DDoS 공격에서 가장 많이 활용되는 기법이다.
- 공격자는 증폭기(amplifiers) 역할을 하는 취약한 네트워크 서비스를 악용하여, 작은 요청으로 매우 큰 응답을 생성하여 피해자를 공격한다.

- 증폭 공격의 과정
	1. 공격자는 UDP 기반의 증폭 취약점이 있는 서비스를 찾는다.<br>
	2. 공격자는 출발지 IP를 피해자의 IP로 위조한 뒤, 해당 서비스에 요청을 보낸다.<br>
	3. 증폭기가 요청을 받으면, 피해자의 IP로 훨씬 더 큰 크기의 응답을 전송한다. <br>
	4. 피해자는 자신이 요청하지 않은 대량의 트래픽을 받아 Denial-of-Service 상태가 된다.

- 공격자가 선호하는 서비스 특징
	- 출발지 IP를 검증하지 않는다.
	- 응답 크기가 요청보다 훨씬 크다(즉, 증폭 계수(Amplification Factor)가 높음).

- 피해 사례
	- PayPal, Spotify, Twitter, Reddit, eBay 등 주요 인터넷 서비스가 DDoS 공격으로 일시적 마비되었다.
	- 2021년에는 2.4 Tbps 규모의 공격이 발생하였다.

- 증폭 공격의 파급력
	- 공격자가 투자하는 트래픽 대비, 피해자가 받는 공격 트래픽이 수십 배에서 수천 배까지 증가할 수 있다.
	- 공격에 사용되는 증폭 벡터가 새롭게 발견될 때마다, DDoS 공격의 규모가 커지는 경향을 보인다.

### 2.3 Amplification Vulnerabilities

- 현재까지 발견된 대부분의 증폭 취약점은 UDP 기반 프로토콜에서 발생하였다.
    - 대표적인 프로토콜: DNS, NTP, SNMP, etc.
- UDP는 비연결형(Connectionless) 프로토콜이므로 출발지 주소 spoofing이 가능하며, 공격자가 이를 악용하여 증폭 공격을 수행할 수 있다.

- 사례: NTP ‘monlist’ 취약점
	- 네트워크 시간 프로토콜(NTP)의 `monlist` 기능을 사용하면, 작은 요청을 통해 최대 4,670배까지 트래픽 증폭 가능하다.
	- 취약점이 보고된 후, 10주 만에 취약한 시스템을 92%까지 줄이는 패치가 적용되었다.

- DNS 증폭 공격 완화 사례
	- DNS 증폭 공격은 Rate Limiting, 메시지 트렁케이션Truncation 등의 기법을 통해 완화된다.
	- 불필요한 기능을 비활성화하거나 패치 적용으로 위험을 줄일 수 있다.

- 증폭 취약점의 특수성
	- 일반적인 보안 취약점(e.g. 메모리 오류, 충돌)과 다르게, 증폭 취약점은 프로토콜의 정상적인 동작을 악용한 것이다.
	- 즉, 버그라기보다는 의도된 기능을 공격자가 악용하는 형태이다.
	- 따라서 기존의 퍼징 도구들은 증폭 취약점을 자동으로 탐색하는 데 적합하지 않다.

## 3. Fuzzing for Amplification


### 3.1 Amplification Fuzzing Challenges

기존 퍼징 기법을 그대로 적용하기 어려운 여러 가지 문제점이 존재한다. 이를 해결하기 위해 AMPFUZZ는 새로운 접근 방식을 도입했다.

- 프로토콜 지식 부족 (Lack of Protocol Knowledge)
	- 네트워크 프로토콜은 각기 다른 형식, 구문, 기능을 가진다 → 기존 연구에서는 특정 프로토콜 사양을 기반으로 퍼징 요청을 생성했다.
	- 문제점:
	    - 모든 프로토콜의 사양을 미리 알 수 없다.
	    - 일부 증폭 벡터는 특정한 구현 방식(예: NTP의 monlist 기능)이나, 잘못된 요청 처리를 악용한다 → 사전 지식이 있으면 탐색이 어렵다.
	- 해결책:
	    - AMPFUZZ는 사전 지식 없이도 증폭 취약점을 탐색할 수 있도록 "프로토콜 독립적(protocol-agnostic)" 퍼징 기법을 도입했다.


- UDP 상태 정보 부족 (Lack of UDP State)
	- UDP는 비연결형 프로토콜이므로, 서비스가 요청을 언제 처리했는지 알기 어렵다.
	- 문제점:
	    - 잘못된 요청은 단순히 무시될 수 있어, 퍼징 도중 피드백을 받기 어렵다.
	    - 응답이 올 때까지 기다려야 하는데, 응답이 지연되거나 오지 않는 경우 불필요한 타임아웃이 발생한다.
	    - UDP는 연결 상태를 유지하지 않으므로, 언제 테스트를 종료해야 하는지 판단하기 어렵다.
	- 해결책:
	    - AMPFUZZ는 UDP-aware 퍼징 기법을 사용하여 서비스가 요청을 수락할 준비가 되었는지, 요청 처리가 끝났는지를 효과적으로 판단한다.

- 새로운 취약점 클래스 (Unexplored Vulnerability Class)
	- 일반적인 취약점(e.g. 충돌, 메모리 오버플로우)과 달리, 증폭 취약점은 단순히 프로그램이 크래시(crash)하는 것이 아니라, 트래픽 증폭 여부가 중요하다.
	- 문제점:
	    - 요청을 보냈다고 해서 반드시 증폭 취약점이 있는 것은 아니다.
	    - 증폭 계수(Amplification Factor)를 극대화할 수 있는 요청을 찾아야 한다.
	- 해결책:
	    - AMPFUZZ는 요청-응답 크기 비율을 최대화하는 방향으로 퍼징을 최적화한다.


### 3.2 Design Overview

AmpFuzz는 기존 Greybox Fuzzing 기법을 UDP 환경에 맞게 개선하여 설계되었다.

*AMPFUZZ는 세 가지 주요 컴포넌트로 구성된다.*
1. 정적 분석기(Static Analyzer) <br>
    - 제어 흐름 그래프(Control Flow Graph, CFG)와 주요 타겟 위치를 추출하여 퍼저(fuzzer)를 가이드한다.<br>
2. 프로그램 계측기(Program Instrumentor) <br>
    - 두 개의 계측 버전 생성:
        - 경량 커버리지 수집 버전 → 새로운 입력이 새로운 코드 경로를 탐색했는지 빠르게 테스트한다.
        - 데이터 흐름 분석 버전 → 입력 데이터가 코드 실행 흐름에 미치는 영향을 분석한다. <br>
3. 퍼저(Fuzzer) <br>
    - CFG와 대상 위치 리스트를 사용하여 입력 변이를 최적화한다.
    - 데이터 흐름 분석 결과를 활용하여 가장 증폭 가능성이 높은 요청을 생성한다.

### 3.3 Protocol-Agnostic Fuzzing

- ==AmpFuzz는 특정 프로토콜에 의존하지 않는다.==
- 네트워크에서 패킷을 전송하는 함수들을 퍼징의 타겟으로 설정하여 자동 탐색한다.
- 네트워크에서 패킷을 수신하는 함수들을 taint sources로 인식하여 입력 변이를 최적화한다.
- 이를 통해, 유효한 응답을 생성할 가능성이 높은 요청을 빠르게 탐색한다.


### 3.4 UDP-Aware Fuzzing

UDP의 특성을 반영하여 서비스가 요청을 받을 준비가 되었는지, 처리가 끝났는지를 인식할 수 있도록 개선한다.

> Table 1: Network function classification

| Class                        | Functions                                                                                                                                       |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| *Source<br>Sink<br>Blocking* | `recv`, `recvfrom`, `recvmsg`, `recvmmsgsend`, `sendfrom`, `sendmsg`, `sendmmsgselect`, `pselect`, `poll`, `ppoll`, `epoll_wait`, `epoll_pwait` |


- 요청 처리 시작 감지
	- 네트워크 함수들을 **세 가지 유형**으로 분류:
	    1. **소스(Source) 함수**: 패킷을 수신하는 함수 (e.g. `recvfrom()`, `read()`). <br>
	    2. **싱크(Sink) 함수**: 패킷을 전송하는 함수 (e.g. `sendto()`). <br>
	    3. **블로킹(Blocking) 함수**: 요청을 기다리는 함수 (e.g. `select()`, `poll()`).

> 서비스가 요청을 받을 준비가 되었을 때:
- `recvfrom()` 같은 소스 함수가 호출되면, 서비스가 요청을 받을 준비가 된 것으로 판단한다.
- 이를 fuzzer에게 신호(shared memory semaphore)로 전달하여 불필요한 타임아웃을 방지한다.

- 요청 처리 완료 감지
	- 요청 처리가 끝난 시점을 판단하는 것은 쉽지 않다. 특히, 멀티스레드 환경에서 어느 시점에 응답이 끝나는지 예측하기 어렵다.
	- 해결책:
	    - Sink 함수를 호출하기 전까지는 요청이 아직 처리 중인 것으로 간주한다.
	    - 특정한 기본 블록이 응답을 생성할 수 있는지 여부를 정적으로 분석한다.
	    - `check-and-terminate` 기능을 추가하여 불필요한 계산이나 I/O 작업을 수행하기 전에 퍼징을 종료한다.

> Example. 네트워크 daemon의 일반적인 실행 흐름
1. 초기화 단계(init) → 소켓을 열고 대기. <br>
2. 루프(loop) 실행 → 패킷 수신 (source `recv()`). <br>
3. 요청 처리(handle) → 응답이 필요한 경우, `send()` 호출 (sink function). <br>
4. 응답을 보내지 않고 로깅할 경우, 로그 기록 후 종료.

*AmpFuzz는 이런 구조를 분석하여, 요청-응답의 흐름을 효과적으로 추적한다.*

### 3.5 Amplification Feedback and Optimization

증폭 벡터를 찾기 위해 BAF(Bandwidth Amplification Factor)를 활용한다.

$$
BAF = \frac{\text{len}(\text{UDP payload from amplifier to victim})}{\text{len}(\text{UDP payload from attacker to amplifier})}
$$

- 즉, *공격자가 보내는 요청 대비 피해자가 받는 응답의 크기 비율*.

- 기존 BAF 계산의 문제점 해결
	- 일부 네트워크 서비스는 0바이트 요청에도 응답을 보내거나 여러 개의 응답 패킷을 보낸다.
	- 이를 반영하기 위해 Ethernet Layer까지 포함한 BAFL2 지표를 도입했다.
	    - UDP 헤더 (8바이트)
	    - IP 헤더 (20바이트)
	    - 이더넷 헤더 & 트레일러 (18바이트)
	    - 이더넷 최소 payload 크기 (46바이트) 적용.

> 수식: L2 레벨에서의 증폭 계수 (BAFL2)

$$
\text{lenL2}(x) = 18 + \max(46, 20 + 8 + \text{len}(x))
$$

$$
BAF_{L2} = \frac{\sum \text{lenL2}(\text{UDP payload fuzz output})}{\text{lenL2}(\text{UDP payload fuzz input})}
$$


- 즉, *요청/응답 크기를 더 정확하게 측정하여 실제 증폭 수준을 평가*.

- 퍼징 피드백 최적화
	- 기존 퍼징 기법은 새로운 코드 경로를 탐색하면 해당 입력을 새로운 seed로 저장한다.
	- AmpFuzz는 여기에 "증폭 계수를 높이는 입력도 추가 저장"하여 증폭 최적화한다.
	- BAFL2 ≤ 1 (즉, 증폭이 없는 입력)도 유지 → 이후 변이(mutation)로 증폭이 증가할 가능성이 있기 때문이다.

- 경로별 증폭 최적화
	- 서비스마다 증폭 벡터가 다를 수 있다 → 특정 Path에서 최대 증폭 계수를 별도로 유지한다.
	- 이를 통해 동일한 서비스에서 여러 개의 증폭 벡터를 독립적으로 탐색한다.


==증폭 계수 극대화 기법 (Amplification Maximization)==

AMPFUZZ는 두 가지 방식으로 증폭 계수를 높인다.

1. 응답 크기를 증가시키는 요청 변이 <br>
    - Dataflow Analysis를 활용하여, 응답 크기에 영향을 미치는 입력 바이트를 식별한다.
    - 특히, `send()` 같은 전송 함수의 길이 인자를 조작하여 증폭 증가 가능성을 높인다.
    - 이를 통해, 응답을 증가시키는 변이를 우선 탐색한다.

2. 최소 요청 크기 탐색 <br>
    - 유효한 요청을 찾으면, 한 바이트씩 줄이면서도 동일한 응답을 유도할 수 있는 최소 요청을 찾는다.
    - 일부 네트워크 프로토콜은 패킷 끝부분(trailing bytes)을 무시하거나, NULL 바이트로 padding하기 때문에 짧은 요청도 유효할 가능성이 있다.

> **결과**
최대 응답을 유도하는 입력을 탐색하면서, 최소 요청 크기로 증폭 계수를 극대화하는 접근 방식을 병행했다.


### 3.6 Implementation

AMPFUZZ는 기존 퍼징 프레임워크인 ParmeSan(퍼징 엔진)과 Angora(데이터 흐름 분석 기반 퍼저)를 확장하여 구현되었다.

- LLVM 기반 정적 분석기 및 계측기
    - 프로그램을 wllvm을 사용해 단일 비트코드 파일로 변환하여 분석.
    - 기존 ParmeSan을 확장하여 UDP-aware 계측 기능 추가.
- 데이터 흐름 추적 확장
    - 기존 ParmeSan은 공유 라이브러리(Shared Libraries) 분석이 불가능.
    - AMPFUZZ는 동적 로드된 코드(e.g., 플러그인 시스템)까지 분석 가능하도록 확장.
    - `dlsym()` 및 `dlopen()` 함수를 hooking → 동적으로 로드된 코드도 CFG 분석 가능.
- fork() 처리
    - 네트워크 데몬(daemon)들은 fork()를 통해 새로운 요청 처리 프로세스를 생성.
    - 기존 퍼징 기법은 부모 프로세스만 추적했으나, AMPFUZZ는 fork된 자식 프로세스까지 추적.
- inetd 서비스 지원
    - 일부 데몬(e.g., `in.tftpd`)은 UDP 소켓을 stdin/stdout으로 사용하는 inetd 기반 서비스.
    - 이를 지원하기 위해 별도의 래퍼(wrapper) 라이브러리 구현.

## 4. Evaluation


### 4.1 Research Questions

AMPFUZZ의 성능을 평가하기 위해 다음과 같은 연구 질문을 설정하였다.
- RQ1: AmpFuzz는 사전 정보 없이 증폭 취약점을 찾을 수 있는가?
    - 기존에 알려진 취약점을 포함한 다양한 네트워크 서비스를 대상으로 테스트.
- RQ2: UDP-aware 퍼징이 정적 타임아웃보다 빠르게 취약점을 찾을 수 있는가?
    - 정적 타임아웃(10ms, 50ms, 100ms, 500ms, 1000ms)과 비교하여 AmpFuzz가 첫 증폭 벡터를 찾는 시간을 측정.
- RQ3: 증폭 최적화(Amplification Maximization) 기법이 순수한 커버리지 기반 네트워크 퍼징보다 더 높은 증폭 계수를 제공하는가?
    - 증폭 최적화 기능이 활성화된 경우와 비활성화된 경우의 최대 BAFL2 값을 비교.


### 4.2 Fuzz Target Selection and On-Boarding

- 대상 서비스 선정
	- SELinux 정책 데이터베이스에서 UDP 포트를 사용하는 프로그램을 찾는다.
	- Debian 패키지 데이터베이스와 교차 검증하여 UDP 기반 네트워크 서비스 71개(61개 패키지) 후보를 선정한다.
	- LLVM 계측이 가능한 20개 서비스(18개 패키지)를 최종 선택한다.

- 특이 사항
	- DNS 서버(bind, dnsmasq)는 계측이 실패하여 제외한다.
	- 대신 knotd, stubby를 포함하여 DNS 증폭 공격 가능성을 평가한다.

- 최종 테스트 환경
	- UDP 포트를 열고 실행 가능한 상태에서 Docker 컨테이너에서 퍼징을 수행한다.
	- 최종 테스트 대상: 28개 네트워크 서비스


### 4.3 Experimental Setup

- Debian: bullseye 기반 Docker 컨테이너에서 실행.
- UDP-aware 설정: 요청 시작 및 종료 타임아웃을 500ms로 설정.
- 각 퍼징 실험을 5회 반복하여 랜덤성 감소.
- Intel Xeon Gold 6230N (2개) / 512GB RAM 서버에서 실행.


### 4.4 (RQ1) Efficacy of Fuzzing for Amplification Vulnerabilities

- 24시간 동안 AmpFuzz 실행  
- 증폭(BAFL2 > 1) 취약점 13개 발견, 반사(Reflection) 취약점(BAFL2 = 1) 6개 발견

### 4.5 (RQ2) Impact of UDP-aware fuzzing

- UDP-aware 설정이 정적 타임아웃보다 빠르게 증폭 취약점 발견  
- 정적 타임아웃 최적값이 서비스마다 달라서 비효율적 
- UDP-aware 설정이 최적 타임아웃을 자동 조정하여 탐색 속도를 향상
- 예외) Knotd, Memcached의 경우 백그라운드 스레드 문제로 탐색 속도 저하

![Figure 4](AmpFuzz_4.png)
UDP-aware 퍼징 vs. 정적 타임아웃에서 첫 번째 응답까지 걸리는 시간 비교 (낮을수록 좋음)
즉, UDP-aware 퍼징과 여러 정적 타임아웃 설정에서 첫 번째 응답을 받기까지 걸리는 시간을 측정한 결과를 비교하며, 더 빠른 응답(낮은 값)이 더 우수한 성능을 의미한다는 뜻이다.


### 4.6 (RQ3) Amplification Maximization

- 증폭 최적화 적용 시 대부분의 서비스에서 더 높은 BAF 계수 달성
- 일부 서비스는 커버리지 기반 탐색만으로도 최대 증폭 벡터 발견 가능
- Memcached, BOS Server 등에서는 요청 크기 최적화를 통해 증폭 계수를 극대화


### 4.7 Case Study: openafs *bosserver*

- OpenAFS의 BOS Server(포트 7007)에서 신규 증폭 벡터 발견
- 특정한 디버깅 패킷 요청 시 312바이트 응답 → BAF = 9.75 (BAFL2 = 4.59)
- OpenAFS 전체 서비스(RX RPC 프로토콜 공유)에서 동일한 취약점이 존재함을 확인
- 인터넷 전체 스캔 결과, OpenAFS BOS Server 약 1,000대, 전체 OpenAFS 서비스 16,000대가 취약
- DDoS 공격에 악용될 가능성이 크며, 보안 패치가 필요하다.

### 4.8 Case Study: Honeypot Synthesis

- 증폭 취약점 공격을 유인하기 위한 허니팟을 자동으로 생성하는 방법 제안
- AmpFuzz가 발견한 요청-응답 쌍을 활용하여 허니팟 동작을 자동 생성
- LLVM 기반 심볼릭 실행(SymCC)으로 요청 조건을 추출하고 파이썬 코드로 변환
- 실제 취약 서비스와 유사한 응답을 제공하며, 공격자의 행위를 관찰 가능
- 기존 수동 분석 대비 빠르고 정확한 허니팟 구축 가능

### 4.9 Comparison with AmpMap

- AmpMap (Moon et al., 2021)과 비교하여 AmpFuzz의 장점을 분석

| 비교 항목      | AmpFuzz              | AmpMap                         |
| ---------- | -------------------- | ------------------------------ |
| 계측         | 필요하다(서비스 내부 분석 가능).  | 필요 없다(실제 서비스 테스트 가능).          |
| 퍼징 방식      | Greybox Fuzzing      | Grammar-based Blackbox Fuzzing |
| 프로토콜 사전 지식 | 필요 없다(프로토콜 독립적).     | 필요하다(문법 기반 요청 생성).             |
| 새로운 취약점 탐색 | 가능(9개 신규 증폭 벡터 발견)   | 불가능(문법 기반이라 예상 범위 내에서만 탐색)     |
| 패킷 변형 기법   | 자동 최적화 가능(증폭 극대화)    | 불가능                            |
| 보안 연구 기여   | 퍼징을 통한 새로운 증폭 취약점 발견 | 기존 프로토콜의 증폭 취약점 확인             |

## 5. Discussion


### 5.1 Evaluation Shortcomings

**(1) LLVM IR 의존**

> 문제점
> - AMPFUZZ는 LLVM IR 기반 계측(Instrumentation)을 사용하므로, **LLVM을 지원하는 프로그램만 퍼징 가능**하다.
> - 평가에서는 wllvm을 사용하여 C/C++로 작성된 네트워크 서비스만 포함한다.
> - 스크립트 언어(예: Perl) 또는 GCC-specific 확장(e.g., 인라인 어셈블리 사용) 서비스는 분석에서 제외되었다.
<br>

- 가능한 해결책
	- 비LLVM 기반 바이너리도 지원하는 계측 기법을 연구한다(e.g., 바이너리 인스트루먼테이션 활용).


**(2) UDP 소켓 사용의 제한**

> 문제점
> - AmpFuzz는 실제 UDP 소켓을 사용하여 입출력을 전달한다.
> - 이 방식의 장점:
> 	- 실제 환경과 동일한 API 동작을 보장한다(`poll`, `select` 같은 소켓 상태 기반 API 포함).
> - 문제점:
> 	- Docker를 사용하여 프로세스를 격리하지만, 일부 저수준 시스템 호출(예: `ioctl`)이 제한된다 → 일부 타겟이 제외된다.
> 	- 하나의 포트/주소에 하나의 소켓만 바인딩 가능하다 → 병렬 퍼징이 어렵다.
<br>

- 가능한 해결책
	- 소켓 API를 hooking하여 실제 소켓을 사용하지 않고 가상화하는 방법을 고려한다.
	- `127.0.0.0/8` 범위 내에서 개별 인스턴스가 서로 다른 주소를 바인딩하도록 설정하여 퍼징 속도 향상이 가능하다.


**(3) Source Addresses 문제**

> 문제점
> - 일부 네트워크 데몬은 로컬 주소에서 온 요청을 무시하거나, 반대로 비로컬 주소에서 온 요청을 차단하는 경우가 있다.
> - AMPFUZZ는 기본적으로 로컬에서 실행되므로 일부 타겟에서는 요청이 정상적으로 처리되지 않을 가능성이 있다.
<br>

- 가능한 해결책
	- 수동으로 SuT(테스트 대상 서비스)의 설정을 점검하여 올바른 주소를 찾는다.
	- AmpFuzz를 확장하여 로컬 및 비로컬 주소를 모두 테스트하는 기능을 추가한다.


### 5.2 Limitations

**(1) 단일 UDP 요청 모델**

- AMPFUZZ는 UDP 기반의 증폭 취약점을 찾는 데 최적화되었다.
	- 단일 UDP 요청을 보내고, 그에 대한 응답 크기를 측정하여 증폭 취약점을 분석하는 방식.
	- 이 방식은 현재까지 알려진 대부분의 증폭 취약점 탐색에 유효하다.

- 그러나 AMPFUZZ가 탐색하지 못하는 증폭 벡터도 존재한다.
	- **TCP 기반 증폭(TCP-based amplification)**:
	    - 일부 증폭 취약점은 TCP를 통해 발생할 수 있다.
	- **다중 요청 기반 증폭(Multi-request amplification)**:
	    - 공격자들이 사전 TCP 요청을 통해 Memcached 서버에 큰 페이로드를 삽입한 후 UDP를 사용하여 증폭을 유발한 사례가 있다.
	- 현재 AMPFUZZ는 UDP 기반, 단일 요청 방식의 증폭 취약점만 탐색 가능하다 → TCP 및 다중 요청 증폭은 분석 범위에서 제외되었다.


**(2) 요청 복잡성**

- 일부 증폭 취약점은 Greybox Fuzzing으로 찾기 어렵다.
	- DNS 증폭 공격:
	    - 유효한 도메인 이름을 알아야만 증폭 요청을 생성할 수 있다.
	    - DNS 증폭 취약점을 찾으려면 전문가 지식 또는 외부 데이터베이스가 필요하다.
	- LDAP 증폭 공격
	    - ASN.1 BER(기본 부호화 규칙) 형식으로 인코딩된 요청을 필요로 한다.
	    - 단순한 퍼징으로는 올바른 요청을 생성하기 어렵다.  

- 해결책
	- 정형화된 프로토콜 명세가 존재하는 경우, 문법 기반 퍼징이 사용 가능하다.
	- 이를 통해 복잡한 요청 형식이 필요한 취약점도 탐색할 수 있다.


**(3) 계측 가능성 (Instrumentation)**

- AmpFuzz는 대상 프로그램이 계측가능하다는 가정을 기반으로 한다.
	- 즉, 대상이 LLVM 기반의 계측 도구로 분석될 수 있어야 한다.  
	
	- 문제점
		- 폐쇄형(Closed-source) 소프트웨어는 계측이 불가능하여 퍼징할 수 없다.
	- 해결책
		- 최근 **바이너리 전용 퍼징(Binary-only Fuzzing)** 기술이 발전하고 있으며, 이를 활용하면 폐쇄형 프로그램에서도 증폭 취약점을 찾을 수 있을 가능성이 있다.


**(4) 대상 서비스 설정**

- AmpFuzz는 각 테스트 대상 서비스(SuT)에 대해 수동 설정이 필요하다.
	- 명령줄 인자(command-line arguments) 및 설정 파일을 수동으로 조정해야 한다.  
	
    - 문제점
		- 일부 네트워크 서비스는 설정 변경에 따라 증폭 계수가 달라질 수 있다.
		- 모든 설정을 자동화할 방법이 없으며, 수동 조정이 필요하다.  
    - 해결책
		- OSS-Fuzz와 같은 방식으로 소프트웨어 개발자가 직접 퍼징 설정을 제공하는 방식 도입한다.
		- 이를 통해 대규모 자동화된 퍼징 환경을 구축할 수 있다.


### 5.3 Active Measurements

- AmpFuzz로 발견된 증폭 취약점이 실제 인터넷에서 얼마나 널리 퍼져 있는지 평가하기 위해 전 세계 인터넷 스캔을 수행했다.
- ==윤리적 가이드라인을 준수==하며, 다음과 같은 조치를 시행했다.
    - 무작위 샘플링된 IP만 스캔하여 특정 네트워크에 부하를 주지 않았다.
    - 각 IP당 단일 요청만 전송하여 서비스 중단을 방지한다.
    - 연구소 차단 목록(블록리스트) 준수하여 요청 거부된 네트워크는 스캔을 제외한다.
    - 윤리 검토 위원회(ERB) 승인 후 실험을 수행한다.
    - OpenAFS 사례 연구에서 사용한 디버그 패킷이 서비스에 영향을 주지 않음을 사전 검증한다.


### 5.4 Coordinated Disclosure

- 연구 결과를 논문으로 제출하기 전에 해당 취약점이 발견된 소프트웨어의 유지보수자(벤더)들에게 사전 연락을 취했다..
- 최소 90일 동안 취약점 비공개 유지하여 개발자가 보안 패치를 적용할 수 있도록 한다 (업계 표준 준수).
- 논문 제출 전 어떤 벤더도 연구 결과 수정(비공개 요청)을 요구하지 않는다.

## 6. Related Work


### 6.1 Amplification DDoS

- 초기 연구: 2001년 Paxson이 UDP 기반 반사(Reflection) 공격 가능성을 발견했다.
- 2014년 Rossow: 14개의 취약한 UDP 프로토콜을 분석하고, 실제 증폭 계수를 측정했다.
- 개별 프로토콜 연구:
    - DNSSEC 도입 후 증폭 영향 연구
    - DNS 쿼리 최적화 공격 연구
    - P2P 네트워크를 이용한 증폭 공격 연구→ Memcached 증폭 공격과 유사.
    - TCP 핸드셰이크 증폭 가능성 분석.
    - IGMP 프로토콜 증폭 공격 연구.
    - BACnet 프로토콜의 증폭 공격 위험성 경고.
- 기존 연구의 한계:
    - 모든 증폭 벡터는 수동 분석(Manual Protocol Review)으로 발견되었다.
    - ==AmpFuzz는 이를 자동화하는 것을 목표로 했다.==
- DDoS 공격 방어 연구:
    - 취약점 발견 후 NTP 증폭 공격 방어율 92% 감소.
    - DDoS 허니팟 연구 → 공격자의 행위를 감시하고, 수사 지원 가능.


### 6.2 Algorithmic Complexity DoS

- 퍼징(Fuzzing)은 리소스를 과도하게 사용하는 입력을 찾아 DoS 공격을 유발하는 취약점을 탐색하는 데 성공적으로 활용되었다.
- SlowFuzz & AFL 기반 퍼징:
    - 비용이 높은 코드 경로를 탐색하여 AC(Algorithmic Complexity) 취약점을 발견했다.
- 바이너리 심볼릭 실행을 활용한 연구 → 더 깊은 실행 경로 탐색 가능하다.
- 마이크로 퍼징(Micro-fuzzing) 기법 연구 → Java 프로그램의 특정 함수 수준에서 AC 취약점 탐색한다.

- AC 버그와 증폭 취약점의 차이:
	- AC 취약점은 해당 시스템 자체를 DoS 상태로 만드는 것이 목표이다.
	- 반면, 증폭 공격은 취약한 시스템을 "중계기(Amplifier)"로 사용하여 다른 시스템을 공격한다.


### 6.3 Fuzzing with Symbolic Execution

- 심볼릭 실행(Symbolic Execution)은 퍼징이 탐색하기 어려운 깊은 코드 경로를 찾는 데 유용하다.
- 하이브리드 퍼징 기법 → 퍼징 + 제약 해결(Constraint Solving)로 새로운 입력을 생성한다.
- 최적화 연구:
    - 단순화된 프로그램에서 퍼징 후 원본에서 실행 가능성 검토
    - 초기 입력(Seed) 생성에만 심볼릭 실행 사용하여 속도 개선
- AMPFUZZ 차별점:
    - 심볼릭 실행을 *퍼징이 아닌 허니팟 생성* 에만 활용


### 6.4 Network Fuzzing

- 기존 네트워크 퍼징 도구들은 주로 **TCP 서비스** 대상.
- UDP 퍼징 도구는 응답을 무시하거나 단순 타임아웃을 사용
- 일부는 사용자가 직접 타겟별 스크립트를 작성해야 한다.
- AmpFuzz 차별점:
    - UDP-aware 퍼징 기법을 통해 기존 방법보다 훨씬 빠르게 증폭 취약점 탐색 가능
    - 크래시 탐색이 아닌 "증폭 취약점 자동 탐색"을 목표로 하는 최초의 퍼징 도구

## 7. Conclusion

- AmpFuzz는 최초의 프로토콜 독립적(Protocol-Agnostic) 증폭 DDoS 취약점 탐색 도구이다.  
- Greybox Fuzzing) 기법을 활용하여, 큰 응답을 유발하는 입력을 자동 탐색한다.  
- UDP-aware 퍼징 기법 도입 → ==프로토콜 상태를 인식하여 퍼징 성능 향상한다.== 
- 실제 네트워크 서비스(28개) 평가 결과:
  - 19개 서비스에서 취약점을 발견했다(13개는 BAFL2 > 1).  
  - 7개의 기존 취약점 재발견, 9개의 신규 취약점을 발견했다.  
- 가장 심각한 신규 취약점(BAFL2 = 4.59)에 대해 인터넷 전역 스캔 수행하여 실제 공격 가능성을 입증했다.
